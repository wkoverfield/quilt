# Live run — L1 disjoint — 2026-06-30

- **Rung:** L1 (disjoint at scale)
- **Agents:** 3 general-purpose coding agents (Claude), running concurrently
- **Seed repo:** one `src/pricing.js` with four independent functions
  (`baseRate`, `discount`, `tax`, and `total` which composes them)
- **Task:** each agent edits a different function in the same file —
  rate-agent: `baseRate` → 120; discount-agent: `discount` → ×0.85;
  tax-agent: `tax` → ×1.07. All three launched at once, sharing one checkout.
- **Conditions:** WITHOUT Quilt (plain git, no coordination) vs WITH Quilt
  (each agent: `start` → `claim src/pricing.js#<fn>` → edit → `commit --mine`).

## Result: both conditions succeeded — for different reasons

| Metric | WITHOUT Quilt | WITH Quilt |
|--------|---------------|------------|
| all features landed | yes (120, 0.85, 1.07) | yes (120, 0.85, 1.07) |
| silent loss | none | none |
| attribution correct | yes (3 authors) | yes (3 authors) |
| commits | 3, one per agent | 3, one per agent |

Both repos ended identical: three commits, each correctly attributed, every
change present. **The honest read: this run did not produce a WITHOUT-Quilt
failure.** Overclaiming that uncoordinated agents always lose work would be wrong,
and this run shows why.

## Why WITHOUT held up (and why that's luck, not a guarantee)

Two things saved the uncoordinated run, neither of them coordination:

1. **The agents' edit tool already does optimistic locking.** Every agent's edit
   that raced another's was *rejected* ("the file changed underneath me"), and the
   agent re-read and re-applied only its own line. That prevented file-level
   clobbers — but it's a property of the agent's tooling, not of git, and an agent
   using `sed`/`cat >`/`echo >>` would have clobbered freely.
2. **The commits happened to stagger.** Each agent ran `git commit -am` at a
   different moment, so each commit captured only its own change. Had two edits
   been uncommitted when one agent ran `git commit -am`, that commit would have
   absorbed both — the classic misattribution. Timing, not design, avoided it.

So WITHOUT Quilt, a clean outcome here was **probabilistic**: it depended on the
edit tool and on commit timing.

## What WITH Quilt did by construction

Each agent claimed its function (all three granted — disjoint, no contention) and
committed with `commit --mine`. Every agent independently reported the same thing:

> "`commit --mine` committed **only** my change and left the other actors'
> concurrent edits in the working tree, even though they were already present."
> — all three WITH agents

That isolation does not depend on timing or on the edit tool. `commit --mine`
takes only the actor's own lines, so the absorption that WITHOUT avoided *by luck*
can't happen here regardless of interleaving. The claims also gave each agent an
explicit, upfront signal that its target was free.

## What this run does and doesn't prove

- **It proves the integration works with real agents.** Three real agents drove
  the full `start → claim → commit --mine` loop concurrently against one checkout
  and produced clean, correctly-attributed history. The orchestrator drop-in
  ([docs/orchestrators.md](../../../docs/orchestrators.md)) is real.
- **It does not prove "uncoordinated agents lose work."** On this easy (disjoint)
  rung, with race-checking edit tools, plain git was fine. Quilt's win here is
  *reliability and explicit coordination*, not a rescued disaster.
- **The reliable failures live in the scripted ladder.** The deterministic
  WITHOUT-Quilt failures — incompatible conflict (L2), dependency cascade (L3),
  refactor underfoot (L4), emergent overlap (L5), absorption under noise (L6) —
  are reproduced in `bench/` and run in CI. This live run is the real-agent
  counterpart to the *L1* rung specifically.

## Follow-up

A live run on a harder rung (L2 same-line conflict, or L3 dependency cascade
where one agent changes a function `total()` depends on) would show a WITHOUT
failure that this disjoint case can't. Worth doing for launch evidence.
